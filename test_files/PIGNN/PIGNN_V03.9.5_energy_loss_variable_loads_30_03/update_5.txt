Why uz Can't Learn (Even With Fz Data)
text

uz_c = 3.69e-4
ux_c = 1.37e-2
ratio = uz_c/ux_c = 0.027

Gradient for pred_raw[:,0] (ux): ∂W/∂pred_ux = Fx × ux_c/E_c → O(1)
Gradient for pred_raw[:,1] (uz): ∂W/∂pred_uz = Fz × uz_c/E_c → O(0.027)
uz gradient is 37× weaker because of the normalization. The optimizer ignores it.

The Fix: Normalize ux and uz by the SAME Scale
Both are translational displacements in meters. Using different scales created the gradient imbalance.

Use ux_c for both ux and uz normalization. Then:

pred_raw[:,0] = ux/ux_c → O(1) at solution
pred_raw[:,1] = uz/ux_c → O(0.027) at solution (small but learnable)
pred_raw[:,2] = θ/θ_c → O(1) at solution (keep separate — different units)

Gradients become:

∂W/∂pred[:,0] = Fx × ux_c/E_c → O(1)
∂W/∂pred[:,1] = Fz × ux_c/E_c → O(1)  ← NOW EQUAL!

Option 1: Per-DOF Decoder Scaling (Try First)
The network outputs O(1) for all DOFs, but uz needs O(0.027). Add a fixed scale factor so the decoder doesn't have to learn the magnitude:

In model.py, add to __init__:

Python

        # Fixed scale so all decoders output O(1) internally
        self.register_buffer('uz_scale', torch.tensor(0.027))  # uz_c / ux_c
In forward, change uz decode:

Python

        ux = self.decoder_ux(z_local)
        uz = self.decoder_uz(z_local) * self.uz_scale   # network outputs O(1), scaled to O(0.027)
        th = self.decoder_th(z_local)
This way the decoder learns O(1) values for uz internally, and the scaling handles the magnitude. Energy gradients flow back through the scale factor correctly.

The Fundamental Answer
Normalization ≠ Energy Contribution
No matter what scale you use, the energy is computed in physical units:

text

W = Σ F_phys × u_phys
The physical reality:

text

ux_physical ≈ 0.0137 m
uz_physical ≈ 0.000369 m
ratio: 37×
The energy contribution scales as u² (strain energy) and u (work):

text

Energy from ux: ~ ux² ~ (0.0137)² ~ 1.88e-4
Energy from uz: ~ uz² ~ (0.000369)² ~ 1.36e-7

Ratio: 1380×
uz contributes 1380× less to the total energy. The optimizer sees that getting ux right reduces the loss 1380× more than getting uz right. So it focuses entirely on ux.

The Catch-22
Normalization	Gradient magnitude	Target value	Loss sensitivity
Each DOF by its own scale	uz gradient 37× weaker	O(1) for uz	uz ignored (weak gradient)
Same scale (ux_c for both)	Equal gradients	uz target = 0.027	uz ignored (tiny loss impact)
Both fail for the same reason: uz physically contributes almost nothing to the energy. Normalization just moves the problem between gradient vs target magnitude.

The model has 301K shared params (encoder+processor) and 25K decoder params. The shared backbone receives gradients from all three decoders:

∂Loss/∂backbone = ∂Loss/∂pred_ux × ∂pred_ux/∂backbone   ← O(1), dominates
